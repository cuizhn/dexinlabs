| # | 第三轮评审意见 | 决策 | 架构影响（4 剪 1 重定位） |

## 1. Context（背景）

### 1.1 v1 → v2 修订说明

ADR0709 v1（初版）于 2026-07-08 提交后通过第一轮架构评审。评审方向正确（去 modules、Data-Render 拆分、Engine 提升应用层），但提出 **8 条具体修订意见**（Core 过大、Pipeline 归属、缺 Shared 层、Utils 分散、Query 顶层不符 Nuxt 官方、Loader 地位过重、Content 命名需彻底移除、最终目录按 10 项重新设计）。本 v2 版 **100% 采纳全部 8 条意见**，并据此重新输出完整决策与迁移方案。

### 1.2 现状（问题来源）

```
app/modules/
└── content/  （单父单子冗余层；16 子目录平铺无分组）
    ├── boot.ts  bootstraps/  cache/  contracts/  core/
    ├── loader/  parser/  queries/  query/  renderer/
    ├── repositories/  schema/  services/  source/
    ├── transformer/  types/  utils/
```

### 1.3 v1 遗留问题（第二轮评审指出）

| # | 评审意见（v1 存在的问题） | 严重度 |
|---|-------------------------|-------|
| R1 | Core 过大：v1 将 Boot + Pipeline 都放 core/，与"核心=长期稳定"定义不符 | 高 |
| R2 | Pipeline 不共享：v1 仍保留 `core/pipeline.ts` 顶层共享管线，实际 DataPipeline 属于 Data、RenderPipeline 属于 Render | 高 |
| R3 | 缺 Shared 层：v1 将 types/schema/utils 分放 Core 和 Data，实际三者都是跨域共享纯类型/纯工具 | 高 |
| R4 | Utils 三域分散：v1 设计 data/utils + render/utils + app/utils 三处并存，长期维护成本 O(n) | 中 |
| R5 | Query 顶层破坏 Nuxt 生态：v1 新建 `app/query/` 装 useXxx Composable，实际属于官方 `app/composables/` | 中 |
| R6 | Loader 被过度设计：v1 将 loaders/ 放 Data 一级公民，实际 @deprecated 历史遗留，不应围绕它 | 低 |
| R7 | "Content" 概念需彻底消失：未来输入类型含 Mermaid/PDF/Video/AI，Content=无信息量概念 | 高（最重要） |
| R8 | 最终顶层目录未按标准：应按 core/data/render/shared + Nuxt 5 目录对齐 | 高 |

### 1.4 刚性约束（必须严格遵守）

- **四不变原则**：API 不变 · Contract 不变 · 调用链不变 · 功能不变
- **调用链严禁修改**：
  - Page → Composable（useAsyncData/useFetch） → Server API → Service → Repository → Drizzle → Neon
  - Parser → Transformer → Renderer
- **严禁破坏 jiagou.md Architecture Priority**：ADR > Spec > Contract > Arch > Implementation

## 2. Decision（决策 · v2 最终版）

### 核心决策 0 · 删除 `app/modules/` 层（方向不变，评审确认保留）

`app/modules/` 仅含 content 一单儿子 = 冗余层。全内容上浮，按能力域直接做 app 顶层目录。

---

### 核心决策 1 · Core 域严格收敛为**三子域**（评审意见 R1 采纳）

**Core 定义 = 真正十年稳定、与具体实现无关的能力**：

```
app/core/
├── contracts/    # ✅ 能力协议（Source/Loader/Query/Parser/Transformer/Renderer）
├── registry/     # ✅ IoC 注册中心（register/get，零具体实现）
└── engine/       # ✅ 组合根 + 统一 Facade（唯一"知道 Data+Render 存在"的地方）
    ├── index.ts         # Engine Facade
    ├── types.ts         # EngineFacade / EnginePlugin 接口（去 Content 前缀）
    └── pipeline.ts      # 【内部薄壳】仅做 "Data vs Render 分流"（不暴露一级概念，回答评审 R2）
```

**v1 → v2 从 Core 迁出的内容**：
- ❌ `boot/` + bootstraps/ → 迁出到独立顶层 `app/boot/`（启动≠核心能力，评审 R1 采纳）
- ❌ `types/` → 迁出到 Shared（见决策 3）
- ❌ `core/pipeline.ts` 共享顶层 → 取消，薄壳下沉到 engine/pipeline.ts，具体 Data/Render Pipeline 归各自领域（评审 R2 采纳）

---

### 核心决策 2 · Boot 独立为顶层 `app/boot/`（评审 R1 采纳 + R2）

**Boot 定义 = Application Startup（应用启动时机问题，非核心能力）**：

```
app/boot/
├── index.ts              # bootEngine() = bootstrapDataSide() → bootstrapRenderSide() → Engine 初始化
├── data/
│   └── database.ts       # Data 侧组件注册：Source=neon-drizzle + Query=lazyQuery
├── render/
│   ├── parser.ts         # Render 侧：注册 MarkdownParser
│   ├── transformers.ts   # Render 侧：按 order 注册 7×Transformer（TRANSFORMER_DEFS 常量表）
│   └── renderer.ts       # Render 侧：注册 VueRenderer
└── infra/                # 【预留空目录】未来 logger/metrics/storage/cache/analytics 基础设施启动
    └── .gitkeep
```

**红线**：同一子目录内 bootstrap 可互 import；跨子目录（data↔render↔infra）**严禁互 import**，只能由 boot/index.ts 做顺序协调。

---

### 核心决策 3 · 新增 Shared 地基层（评审意见 R3 + R4 采纳）

**Shared = 最底层地基土，比 Core 还稳定。纯类型 / 纯 DTO 形状 / 纯工具函数。无业务逻辑、无状态、零反向依赖。**

```
app/shared/
├── types/     # 领域实体类型（Interface 集合）：Course / Chapter / Lesson / Exercise / Asset
│              # 元信息类型：TocEntry / HeadingInfo / LinkInfo / Reference
│              # 未来扩展：User / Video / PDF / MermaidDiagram / AIParagraph
├── schema/    # DTO 投影形状（Repository 查询结果行类型）：
│              # ChapterListByCourseRow / LessonListByChapterRow / ExerciseListByChapterRow
│              # Repository Filters / ListOptions 选项类型
└── utils/     # 统一工具桶（一桶天下，评审 R4 采纳：90% 工具全放这）
               # normalizeSlug / sha1Hash / deepEqual / compact / groupBy / uniqBy / pick / omit
               # formatDate / normalizeUrl / escapeHtml / ORDER_TRANSFORMER 常量表 / 正则表
```

**Shared 四不红线**：
1. 不 import Core（Contracts/Engine/Registry）——Shared 是地基，Core 是宪法
2. 不 import Data / Render——禁止触碰实现层
3. 不含任何 if/else 业务判断（纯函数 / 常量 / 类型）
4. 不含运行时单例状态（无 new Map、无全局缓存）

**Utils 白名单（评审 R4：防分散）**：
- **默认放 shared/utils/**（90% 工具）
- **放 data/_internal/_utils/** 的唯一条件**：含 DB/SQL 专属概念（如 `buildWhereClauseFromFilters()`）
- **放 render/_utils/** 的唯一条件**：含 Markdown/AST 专属概念（如 `walkAstNodes()`、`buildTocTree()`）

---

### 核心决策 4 · Query 顶层 → 回归 Nuxt 官方 `app/composables/`（评审 R5 采纳）

**论证**：useCourse / useChapter / useLesson / useExercise = 100% 标准 Vue Composable，返回 useAsyncData 结果形状，与通用 useXxx 无职责差异 → **无正当理由新建 `query/` 顶层破坏官方约定**。

迁移动作：
```
app/modules/content/query/useCourse.js    → app/composables/useCourse.js
app/modules/content/query/useChapter.js   → app/composables/useChapter.js
app/modules/content/query/useLesson.js    → app/composables/useLesson.js
app/modules/content/query/useExercise.js  → app/composables/useExercise.js
```

**与 Data 侧 `data/queries/` 的命名去歧义（v1 原 `query/` vs `queries/` 混淆根治）**：
| 目录 | 职责 | 使用方 |
|------|------|-------|
| `app/composables/`（Nuxt 官方） | SSR 组合式 useXxx，内调 `useFetch('/api/...')`，**含 Vue/Nuxt API** | Pages / Components（auto-import） |
| `app/data/queries/`（仅 lazyQuery.ts） | QueryContract Facade，**纯 TS，禁止任何 Vue/Nuxt API**，给 Engine boot 注册默认 Query 时用 | 仅 core/engine.boot / boot/data/database |

Nuxt config 变更：删除 `imports.dirs` 中 `'~/modules/content/query'` 条目 → **Nuxt 默认扫 composables/，零新增配置**。

---

### 核心决策 5 · Loader 下沉 Data 内部，不再是一级公民（评审 R6 采纳）

Loader 被标记 `@deprecated`（历史遗留，未来 Services 完全接管后删除），**不应被围绕设计**。v2 将其**打入二级冷宫**：

```
app/data/
├── （5 个一级公民，严格不多不少）
│   ├── sources/        Data 输入（SourceContract 实现）
│   ├── repositories/   DAL
│   ├── services/       业务聚合（长期核心）
│   ├── queries/        QueryContract Facade
│   └── pipeline.ts     Data 内部执行管线
│
├── cache/              可选二级（Data 专属缓存）
│
└── _internal/          🔒 内部实现区，**红线禁止外部 import**（加下划线前缀一眼识别）
    ├── loaders/        @deprecated：2 迭代后删除
    │   ├── course.ts / chapter.ts / lesson.ts / asset.ts
    └── _utils/         Data 专用 SQL 工具（评审 R4 Utils 白名单剩下的 10% Data 专属）
```

**Data → Loader 红线**：
- `server/api/**` / `app/composables/**` / `app/pages/**` / `app/render/**` → 严禁 import `data/_internal/**`
- Code Review 直接打回

### 核心决策 6 · 架构层面**彻底移除 "Content" 命名**（评审 R7，最重要的决策）

**为什么消失**：未来系统输入类型含 Markdown / JSON / Exercise / Mermaid / Diagram / AI-Generated / Video / PDF / …——任何数据都可以是"内容"，Content = 零信息量。真正稳定的三域是 **Data（获取数据）/ Render（渲染东西）/ Core（协调两者）**，与内容无关。

**重命名映射表（8 项，含兼容期策略）**：

| 旧命名（含 Content） | 新命名 | 所在 | 兼容期策略 |
|---------------------|-------|------|-----------|
| `ContentEngineFacade` interface | `EngineFacade` | `core/engine/types.ts` | 立即改，无兼容需求（仅 TS 类型名） |
| `ContentEnginePlugin` interface | `EnginePlugin` | `core/engine/types.ts` | 立即改 |
| `bootContentEngine()` fn | `bootEngine()` | `boot/index.ts` | 旧名 re-export 别名保留 2 周 |
| `getEngine()` fn 名字 | 保持（本来就无前缀） | `core/engine/index.ts` | 不动 |
| `plugins/content-engine.client.js` 文件名 | `plugins/engine.client.js` | `app/plugins/` | 文件名直接改，Nuxt 按文件名后缀自动识别环境 |
| `plugins/content-engine.server.js` 文件名 | `plugins/engine.server.js` | `app/plugins/` | 同上 |
| `provide('contentEngine', engine)` NuxtApp 注入名 | 新增 `provide('engine', engine)`（主）；保留 `provide('contentEngine', engine)` 兼容 7 天 | `plugins/engine.*.js` | 兼容期过后删除 contentEngine 注入 |
| `provide('contentQuery', ...)` 注入 | **删除**，合并到 `$engine.query` 命名空间 | `plugins/engine.*.js` | 如使用方存在老代码，用旧名 re-export 指向 engine.query，7 天后删除 |
| 别名 `@modules/content/*` | 删除 → 替换为 `@core / @boot / @data / @render / @shared` 五别名 | `nuxt.config.ts` + `tsconfig` | 旧别名做 1~2 周 barrel 兼容文件 `app/_legacyModulesContent.ts` |

**例外（不移除 Content 的场合）**：
- 具体输入类型名保留：`MarkdownRenderer`、`MarkdownParser`、`MermaidTransformer`、`ExerciseJSONParser` 等。名字精准 = 具体格式，不删。
- 产品级文档/对外宣传仍可用"德鑫实验室内容平台"——仅代码命名+架构层消失 Content。

## 3. 目录职责重新定义（v2 · 三问法：做什么 / import 什么 / 谁 import 它）

### 3.1 顶层总览

```
app/
├── core/          ① CORE DOMAIN：Contracts 协议 + Registry IoC + Engine 组合根（三子域，极轻）
├── boot/          ② BOOT DOMAIN：Application Startup（从 Core 迁出，独立启动域）
├── data/          ③ DATA DOMAIN：唯一职责 = 提供结构化数据（5 一级公民 + cache + _internal）
├── render/        ④ RENDER DOMAIN：唯一职责 = 将输入渲染为 VNode/HTML（Parsers→Transformers→Renderers）
├── shared/        ⑤ SHARED DOMAIN：地基（Types 类型 + Schema DTO 形状 + Utils 纯工具）
│
├── composables/   Nuxt 官方：useXxx SSR Composable（原 content/query/ → 回归官方）
├── components/    Nuxt 官方：UI 组件（app/home/course/exercise 四域）【原封不动】
├── pages/         Nuxt 官方：文件系统路由【原封不动】
├── layouts/       Nuxt 官方：布局（default.vue）【原封不动】
├── plugins/       Nuxt 官方：engine.client.js / engine.server.js（去 Content 前缀）
├── assets/        Nuxt 官方：CSS / 图片【原封不动】
│
└── app.vue        根组件【原封不动】
```

### 3.2 Core 域三子域详解

#### 3.2.1 core/contracts

| 三问 | 答案 |
|------|------|
| **做什么** | 定义全部能力协议：SourceContract / LoaderContract / QueryContract / ParserContract / TransformerContract / RendererContract + 契约断言 assertContract() + 聚合导出 data.ts / render.ts |
| **import 什么** | 仅 `@shared/types`（协议签名需要领域类型）。**严禁 import 任何实现层（Repo/Parser 等）** |
| **谁 import 它** | Data 实现（implements Source / Query）、Render 实现（implements Parser / Transformer / Renderer）、Core.Registry（声明注册函数签名）、Engine（声明 Facade 类型） |

#### 3.2.2 core/registry

| 三问 | 答案 |
|------|------|
| **做什么** | IoC 注册中心：`registerSource / getSource`、`registerParser / getParser` ... 系列 register/get；内部 Map 存组件实现 |
| **import 什么** | 仅 `@core/contracts/*`（注册函数的参数签名需要契约类型）。**严禁 import 任何具体 Repository / Parser 实现**——实现由 bootstraps 通过参数传入。 |
| **谁 import 它** | boot/data/*.ts + boot/render/*.ts（注册组件时调 register*）；core/engine（get* 拿到实现做编排） |

#### 3.2.3 core/engine（含内部薄壳 pipeline.ts）

| 三问 | 答案 |
|------|------|
| **做什么** | 组合根 + 统一 Facade。对外暴露 `EngineFacade = { data: DataQueryAPI, render: RenderAPI, query: CompatAPI, use(plugin): void }`。内部 `engine/pipeline.ts` 是**薄壳**，只做 "operation ∈ Data？→ data/pipeline；renderContent？→ render/pipeline"，不做具体执行。 |
| **import 什么** | `@core/contracts`、`@core/registry`、**并且是唯一允许同时 import Data + Render 入口文件的地方**（组合根特权）。Engine 也 import @shared/types @shared/utils。 |
| **谁 import 它** | plugins/engine.*.js（NuxtApp.provide `$engine`）；boot/index.ts（bootEngine() 初始化 Engine Facade）。Render 侧主题组件 MarkdownRenderer.vue 允许通过 Engine 调 `engine.renderContent`。 |

**Core 内部依赖单向性红线**：
```
contracts ← registry ← engine
  ↑                      ↓
  └──────── shared ──────┘
```

### 3.3 Boot 域详解（独立顶层，不再属于 Core）

| 子目录 | 做什么 | import 什么 | 谁 import 它 |
|--------|-------|------------|-------------|

## 4. Rationale（权衡 · v2）

### 4.1 为什么 Core 必须极轻（仅 Contracts / Registry / Engine 三）

- Core = 十年不变的宪法。Boot / Pipeline 会随功能增加（新增 Parser / 新 Source）被频繁改动。频繁改动的东西**绝对不能放 Core**——频繁改动的"Core"=虚假的 Core。
- 从"可替换性"验证：如果未来想把 Nuxt → Next / Remix，Contracts / Registry / Engine 三核心可以一行不改带到新框架；Boot / Startup 必须重写。证明 Boot 不属于 Core。

### 4.2 为什么 Pipeline 必须归属各自领域而非共享顶层

- DataPipeline 的执行链是 Source→Repository→Services，用到 SQL / 连接串 / 事务。
- RenderPipeline 的执行链是 Parser→Transformer×N→Renderer，用到 Markdown AST / VNode。
- 两条链没有共享的代码结构（唯一"共同点"=都是 Pipeline 这个英文单词）。硬合并到 `core/pipeline.ts` 共享顶层 = 为了"名字好看"强行制造抽象，违反"目录服务职责"原则。

### 4.3 为什么 Shared 必须单独一层，不能混 Core / Data

- Core 是"协议"（How），Shared 是"数据形状"（What）。协议可以引用 What，但 What 不能引用协议。层级 = Shared 比 Core 更深。
- 如果把 types/schema 放 Data，Render 想知道 `ChapterListByCourseRow` 形状就得 import Data/* = 直接跨域违规（Data↔Render 红线）。放 Shared = Render/Data 都 import Shared，方向单向合法。

### 4.4 为什么 Nuxt 官方目录必须严格遵守（不用自定义 query/ 顶层）

- Nuxt ecosystem 约定：任何 useXxx 文件放 composables/，IDE 插件、静态分析、第三方组件库都基于这个假设做扫描。自定义 query/ = 破坏所有下游约定，团队新人 100% 会"怎么 useChapter 找不到"的问题。
- 调用链的顶层 Composable 层属于 Nuxt 运行时概念，与 Engine 平级而非 Engine 子域，放 Nuxt 官方目录正确。

### 4.5 为什么 Content 命名必须彻底从架构消失

- 命名是边界：ContentEngine 暗示"Engine 服务于 Markdown 内容"，未来加入 VideoRenderer / PDFParser 时，代码命名就产生认知摩擦："为什么 VideoRenderer 放在 Content 下面？它不是 Markdown。"
- 去掉 Content 后命名直接无摩擦：`Engine` 协调一切数据源与渲染器，`render/theme/MermaidRenderer.vue` = 任何人一眼知道在架构中的位置。

### Phase 1 · 文件系统移动（约 40 动作，~2h）

动作模式（详见附录 B 精确 40 步清单，实施前展开）：
1. **先建壳**：创建 `app/core/{contracts,registry,engine/}` + `app/boot/{data,render,infra/}` + `app/data/{sources,repositories,services,queries,cache,_internal/{loaders,_utils/}}` + `app/render/{parsers,transformers,renderers,components,theme,_utils/}` + `app/shared/{types,schema,utils/}` 共约 20 个空目录。

   - Engine Facade + Plugin 类型若写在 engine.ts 内 → 抽离到 `core/engine/types.ts`
4. **Boot 独立顶层**：
   - `boot.ts` → `boot/index.ts`（改名）
   - `bootstraps/database.ts` → `boot/data/database.ts`
   - `bootstraps/{parser,transformers,renderer}.ts` → `boot/render/` 下同名
   - 建 `boot/infra/.gitkeep`
5. **Data 上浮 & Loader 内部化**：
   - 旧 `core/dataPipeline.ts` → `data/pipeline.ts`
   - `source/` → `data/sources/`（source → sources）
   - `repositories/` → `data/repositories/`
   - `services/` → `data/services/`
   - `queries/` → `data/queries/`
   - `cache/` → `data/cache/`
   - `loader/` → `data/_internal/loaders/`（加 @deprecated JSDoc）
6. **Render 上浮**：
   - 旧 `core/renderPipeline.ts` → `render/pipeline.ts`
   - `parser/` → `render/parsers/`
   - `transformer/` → `render/transformers/`
   - `renderer/vueRenderer.js` → `render/renderers/vueRenderer.js`
   - `renderer/components/` → `render/components/`
