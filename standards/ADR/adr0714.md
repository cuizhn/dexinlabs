# 唯一建议调整（建议 V2 即完成）

## Page 返回类型不要放在 models 中

目前新增：

```text
models/

    LessonPage
    ChapterPage
    CoursePage
```

建议调整为：

```text
content-engine/

    models/
        Course
        Chapter
        Lesson
        Exercise

    dto/
        LessonPage
        ChapterPage
        CoursePage
```

原因如下：

### 1. models 表示领域模型（Domain Model）

例如：

```text
Course
Chapter
Lesson
Exercise
```

它们对应的是系统中的业务实体，也是数据库、Repository、Service 的核心对象。

它们具有长期稳定性。

---

### 2. LessonPage 并不是领域模型

例如：

```text
LessonPage

=

Lesson
Chapter
Course
PreviousLesson
NextLesson
```

它只是为了页面返回方便而组合出来的数据。

属于：

> Data Transfer Object（DTO）

而不是：

> Domain Model

因此放在 models 中会混淆两者职责。

---

### 3. 后续扩展更加清晰

未来很可能继续出现：

```text
SearchResultDTO
AdminLessonDTO
LessonTreeDTO
CourseOutlineDTO
LessonEditorDTO
```

如果全部放进 models：

```text
models/

Course
Chapter
Lesson
Exercise
LessonPage
SearchResult
LessonTree
AdminLesson
...
```

最终 models 会越来越混乱。

如果现在建立 dto 目录，后续扩展会保持清晰。

建议目录：

```text
content-engine/

models/
dto/
queries/
services/
```

# 修订记录 · DTO 目录重构

> 日期：2026-07-14
> 依据：评审意见 §唯一建议调整

## 问题

Page 返回类型（`LessonPage`、`ChapterPage`、`CoursePage`）最初放在 `models/` 目录中，混淆了：

- **Domain Model**（领域模型）：`Course`、`Chapter`、`Lesson`、`Exercise` —— 稳定的业务实体
- **DTO**（Data Transfer Object）：`LessonPage`、`ChapterPage`、`CoursePage` —— 页面级临时组合数据

## 解决方案

创建独立的 `dto/` 目录，将 Page 类型迁移过去：

### 目录结构变更

**之前：**
```text
content-engine/
    models/
        Course
        Chapter
        Lesson
        Exercise
        LessonPage      ← 错误：DTO 混入模型
        ChapterPage     ← 错误：DTO 混入模型
        CoursePage      ← 错误：DTO 混入模型
```

**之后：**
```text
content-engine/
    models/
        Course
        Chapter
        Lesson
        Exercise
    dto/
        LessonPage      ← 正确：DTO 独立存放
        ChapterPage     ← 正确：DTO 独立存放
        CoursePage      ← 正确：DTO 独立存放
    queries/
    services/
```

### 文件变更

| 文件 | 变更 |
| --- | --- |
| [dto/index.ts](file:///C:/Users/cui/Documents/www/dexinlabs/app/core/content-engine/dto/index.ts) | **新增**：定义 `LessonPage`、`ChapterPage`、`CoursePage` |
| [models/index.ts](file:///C:/Users/cui/Documents/www/dexinlabs/app/core/content-engine/models/index.ts) | **移除**：删除 Page 类型定义 |
| [content-engine/index.ts](file:///C:/Users/cui/Documents/www/dexinlabs/app/core/content-engine/index.ts) | **调整**：从 `./dto/index` 导出 Page 类型 |

### 向后兼容性

- `@ce` 导出的类型名不变，外部引用无需修改
- Service 层使用内联类型，不受影响
- API 层使用 Service 返回值，不受影响
- Composable / Page 层使用 API 返回值，不受影响

## 设计原则

**models/** 仅存放领域模型（Domain Model）：

- 对应系统中的业务实体
- 具有长期稳定性
- 是数据库、Repository、Service 的核心对象

**dto/** 存放数据传输对象（Data Transfer Object）：

- 为特定场景（如页面展示）组合的数据结构
- 可能随业务需求变化
- 不直接映射数据库表

**后续扩展**：未来新增的 `SearchResultDTO`、`AdminLessonDTO`、`LessonTreeDTO` 等应统一放在 `dto/` 目录下，保持架构清晰。

# ADR0714 V2 修订 · Page Composable 与 Entity Composable 分离

## 背景

Content Engine V2 已完成以下职责划分：

* Repository：数据库访问（CRUD）
* Service：业务规则与数据组合
* API：HTTP 层
* Composable：Vue/Nuxt 适配层
* Page：页面渲染

目前 Service 已经新增：

```text
getCoursePage()
getChapterPage()
getLessonPage()
```

Composable 建议同步升级，使整个 Content Engine 从 Service 到 Composable 保持一致的语义。

---

# 一、Composable 分为两类

## 第一类：Entity Composable（实体）

负责获取单个业务实体。

仅供：

* Admin 后台
* 编辑器
* CRUD
* 表单编辑
* 数据维护

使用。

对应：

```text
useCourse()
useChapter()
useLesson()
```

返回：

```text
Course

Chapter

Lesson
```

不负责任何业务组合。

例如：

```ts
const { lesson } = await useLesson(slug)
```

返回：

```ts
lesson
```

而不是：

```ts
lesson
chapter
course
```

它始终代表：

> 一个 Lesson 实体。

---

## 第二类：Page Composable（页面）

负责获取页面完整数据。

仅供：

前台页面。

对应：

```text
useCoursePage()
useChapterPage()
useLessonPage()
```

分别对应：

```text
CourseService.getCoursePage()

ChapterService.getChapterPage()

LessonService.getLessonPage()
```

Service 与 Composable 保持一一对应关系。

---

# 二、Page Composable 返回方式

不要返回：

```ts
const { page } = await useLessonPage(slug)

page.lesson
page.chapter
page.course
```

也不要返回：

```ts
const { lessonPage } = await useLessonPage(slug)
```

原因：

Page 已经是页面入口。

再次增加 page 对象没有意义。

---

建议直接展开返回页面真正需要的数据。

例如：

```ts
const {
    lesson,
    chapter,
    course,
    previousLesson,
    nextLesson,
    loading,
    error,
    refresh
} = await useLessonPage(slug)
```

其中：

```text
lesson
chapter
course
previousLesson
nextLesson
```

来自：

```text
LessonService.getLessonPage()
```

Composable 负责把 Service 返回结果拆开，并转换成 Vue 响应式对象。

---

Chapter 页面：

```ts
const {
    chapter,
    course,
    lessons,
    exercise,
    previousChapter,
    nextChapter
} = await useChapterPage(slug)
```

---

Course 页面：

```ts
const {
    course,
    chapters
} = await useCoursePage(slug)
```

这样页面无需：

```text
page.course

coursePage.course

lessonPage.lesson
```

而是直接使用真正需要的数据。

---

# 三、Composable 职责

Page Composable 只负责：

* useAsyncData
* $fetch
* loading
* error
* refresh
* SSR
* Hydration
* Nuxt Cache
* 将 API 返回对象转换为 Vue 响应式数据
* 将页面数据拆分为易于使用的字段

禁止：

* 查询多个 API
* 编写业务规则
* 组合业务数据
* 自行计算上一课、下一课
* 调用 Repository

所有业务组合全部来自对应 Service。

---

保持：

Service 继续负责：

* 调用多个 Repository

* 返回页面完整数据

Composable 不参与任何业务。

Repository 继续作为唯一数据访问入口。

负责：

* CRUD
* 查询
* 持久化

不承担任何页面逻辑。

# 七、V2 最终目标

完成本次调整后，Content Engine V2 将形成统一的调用规范：

```text
Vue Page
        │
        ▼
useLessonPage()
        │
        ▼
LessonService.getLessonPage()
        │
        ▼
LessonRepository
        │
        ▼
Database
```

后台管理则采用另一条路径：

```text
Admin
        │
        ▼
useLesson()
        │
        ▼
LessonService.getBySlug()
        │
        ▼
LessonRepository
        │
        ▼
Database
```

最终形成两套清晰且互不混淆的接口：

* **Entity 接口**：服务后台管理、编辑器、CRUD，始终返回单个领域实体。
* **Page 接口**：服务前台页面，始终返回页面所需完整数据，并由 Composable 直接解构为页面可用字段。

这样 Service 与 Composable 命名完全一致，前后台职责明确，页面代码简洁，Content Engine 的分层也保持长期稳定。

## 六、设计原则落地

| 类别 | Composable | 返回 | 使用场景 |
| --- | --- | --- | --- |
| **Entity** | `useCourse()` / `useChapter()` / `useLesson()` | 单个领域实体 | Admin、编辑器、CRUD、表单 |
| **Page** | `useCoursePage()` / `useChapterPage()` / `useLessonPage()` | 页面完整数据（展开字段） | 前台页面渲染 |

Page Composable 遵循以下约束：
- ✅ 仅调用单个 API
- ✅ 使用 `useAsyncData` + `$fetch`
- ✅ 返回 `loading` / `error` / `refresh`
- ✅ 将 API 结果解构为页面可用字段
- ✅ 不编写业务规则
- ✅ 不组合多个接口
