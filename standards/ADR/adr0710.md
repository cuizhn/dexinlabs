## 1. Context（背景）

### 1.1 现状（v3 遗留的 4 大架构问题，触发本轮 v4 调整）

```text
【根目录 Engine 堆积风险】
dexinlabs/
├── markdown-engine/     ← v3 独立引擎在根目录（今天 1 个，明天 content-engine、search-engine、asset-engine 全堆根目录，维护成本 O(n²)）
├── content-engine/      ← v3 计划新建的独立 Engine（如果也放根目录，1 个业务 1 个根目录 Engine 是不可持续模式）
├── server/database/     ← v3 Drizzle 在 server/ 下，业务层可 `from '../server/database'` 直接裸连 Drizzle，破坏依赖倒置
└── app/
    ├── boot/            ← v3 Boot 聚合层：getContentEngine/createMarkdownEngine/ensureDrizzle 硬写死总 Facade → 两 Engine 绑定
    ├── data/            ← v3 services + repositories 混在同一层级 data/ 内（Content Engine 的 Service 与 DB 实现职责未分离）
    ├── render/          ← v3 薄 Vue 适配层（OK）
    └── modules/         ← v3 仅存空壳，未来业务模块无处安置（需与 core/ 并列）
```

### 1.2 4 条刚性红线（v4 必须严格落地，违反即架构失败）

1. ❌ **禁止 Application → Drizzle 裸引用**：`from 'drizzle-orm*'` / `'drizzle-kit'` 只允许出现在 2 个范围 —— `app/core/database/**`（含所有 repos）+ 根 `drizzle.config.ts`（构建工具配置）
2. ❌ **禁止 Page → Repository**：页面层必须走 Content Engine Facade 或 Service，禁止直接 new Repository
3. ❌ **禁止 Component → Database**：组件层只能读 NuxtApp provide 的 `$content/$markdown/$services`
4. ❌ **禁止 Engine 互相引用**：Content Engine 内部不能出现 `@me`；Markdown Engine 内部不能出现 `@ce / Course / Lesson / Chapter / Repository / Service`

## 2. Decision（决策 · v4 终稿 · 分 Round A / Round B）

### 核心决策 A · `app/core/` 五域统一归位（对应评审 Round A 全部意见）

#### A.1 原目录 → 新位置物理迁移 5 张映射（四不变原则：API/Contract/调用链/功能全不变）

| 原路径（迁移前存在） | 迁入 `app/core/` 后路径 | 迁入理由 |
|---|---|---|
| 根 `markdown-engine/` 整个目录（src/ + tests/ + ADR/DESIGN/SPEC 3 文档） | `app/core/markdown-engine/` | Markdown 是纯技术渲染能力，属 Core |
| 根 `content-engine/`（models/ + queries/ + facade index.ts） | `app/core/content-engine/`（models/queries/index 直接保留） | 内容组织/查询/模型属 Core 能力，4 个 Services 随后合并入本目录 |
| `app/data/services/*`（CourseService/ChapterService/LessonService/ExerciseService 4 文件 + index barrel） | `app/core/content-engine/services/*` | 4 个 Service 归属 Content Engine（内容能力），不应该挂在泛化的 data/ 下 |
| `app/data/repositories/*`（5 个 Repository 实现 + index barrel） | `app/core/database/repositories/*` | Repositories 是 Database 层具体实现（Drizzle API 消费者），归属 DB |
| `server/database/*`（connection.ts + schema.ts + migrations/ + index barrel） | `app/core/database/*` | Drizzle/ORM 属 Infrastructure，不进 server/（server/ 仅 API 路由 + Nitro Utils） |

#### A.2 别名重排 3 配置文件（同步删除 1 个旧别名 @data，新增 1 个别名 @core）

| 配置文件 | 变更项 |
|---|---|
| [nuxt.config.ts](file:///C:/Users/cui/Documents/www/dexinlabs/nuxt.config.ts#L56-L80) alias 顶栏 + vite.resolve.alias | **新增**：`@core → app/core`；**修正**：`@me → app/core/markdown-engine/src` / `@ce → app/core/content-engine`；**删除**：`@data → app/data` |
| [tsconfig.json](file:///C:/Users/cui/Documents/www/dexinlabs/tsconfig.json#L13-L27) compilerOptions.paths | **新增双 entry**：`@core` + `@core/*`；**修正双 entry**：`@me/@me/*`、`@ce/@ce/*` 指向 app/core 下新路径；**删除双 entry**：`@data/@data/*` |
| [drizzle.config.ts](file:///C:/Users/cui/Documents/www/dexinlabs/drizzle.config.ts)（根保留） | **修正**：`schema: './app/core/database/schema.ts'` / `out: './app/core/database/migrations'`（原 server/database 对应项） |

#### A.3 Repository 接口类型暴露通道（Content Engine → Database 唯一合法路径）

Content Engine（4 个 Services + facade）**只能经如下两条路径取 Repository 接口/实例**，禁止直接 drizzle：
- **类型消费**：`import type { CourseRepository, ChapterListByCourseRow } from '@core/database/repositories'`
- **实例消费**：`import { courseRepository, chapterRepository } from '@core/database/repositories'`
→ content-engine 自身内嵌的 `repositories/` 空 barrel 文件在 v4 **已物理删除**，切断任何可能的"Content Engine 自产自销 Repository 实现"的错误模式。

### 核心决策 B · Boot 聚合层 100% 删除（对应评审 Round B 全部意见）

#### B.1 删除项 × 替代方式 3 张对照表（Engine 独立 + Application 组合原则）

| 删除对象 | 存在理由（v3 时的考虑） | v4 替代方案 |
|---|---|---|
| **整个 `app/boot/` 目录**（仅 index.ts 1 文件 · 73 行） | v3 认为需要一个"唯一组合根"统一 registerXxx → bootContentEngine() 返回总 Facade | **不需要总组合根**：两 Engine 各自有 `getContentEngine()`（@ce 导出）+ `getEngine()`（@me 导出），Application 自己按需组合 |
| **`@boot` 别名**（nuxt.config 2 处 + tsconfig 2 行） | 供 plugins/engine.*.js 写 `import { ensureDrizzle } from '@boot'` | 删除别名；DB 初始化直接在 2 个 plugins 内联 `await import('@core/database').catch(() => {})`（动态 import + try/catch 语义和 ensureDrizzle 100% 等价） |
| **boot 转出口 API 清单**：getContentEngine/createContentEngine/queries/4 Services/5 Repos/getMarkdownEngine/createMarkdownEngine + 所有类型 | v3 意图"Application 只要 import '@boot' 一手全包" | Application 直接从 2 个 Engine 公开 API 取：`import { X } from '@ce'` / `import { Y } from '@me'`；类型也直接从原 Engine 取，boot 转出口 100% 重复，全部删除无影响 |

#### B.2 Application 编排层标准写法（v4 强制模式）

两 Engine **唯一合法的组合位置** = plugins / pages / server API / composables / Vue Components。以 2 个 plugins 作为标准范例：

```js
// app/plugins/engine.server.js （client.js 完全同构）
import { defineNuxtPlugin } from '#imports'
// 两 Engine 分别 import，互不干涉
import {
  getContentEngine, chapterService, lessonService,
  chapterRepository, assetRepository, queries
} from '@ce'                    // ← Content Engine 自己的公开 API
import {
  getEngine as getMarkdownEngine, renderToHTML, renderToVNode
} from '@me'                    // ← Markdown Engine 自己的公开 API

export default defineNuxtPlugin(async () => {
  // Application 层自行决定启动顺序（不依赖 Boot 层写死）
  await import('@core/database').catch(() => {})
  const content  = getContentEngine()
  const markdown = getMarkdownEngine()
  // Application 层自行 choose 要 provide 哪些能力给全局
  return { provide: { content, markdown, services:{...}, repositories:{...}, queries, renderToHTML, renderToVNode } }
})
```

→ 未来若新增"Audio Engine / AI Engine / Search Engine"，Application 层直接 `import '@audio' / '@search'` 自由组合，**完全不需要再改一个所谓的 boot 总启动层**——这是 v4 架构相较 v3 最大的演进能力。

## 6. 架构红线长期固化（v4 起所有后续 RFC / ADR 必须遵守）

本轮 v4 两大决策形成的 4 条架构禁令**视为项目宪法**：所有后续新增 Engine（Search/Audio/AI）、新增业务模块（Course/Practice/Review）、新增基础设施（Cache/Queue/Storage）都必须遵守，否则架构评审直接否决。

```
┌──────────────────────────────────────────────────────────────────────────────────┐
│ 1. ❌ Application（Page/Component/Composable/Plugin） → Drizzle/ORM 直接引用      │
│        → 必须经 Content Engine Facade / Service，Database 只在 app/core/database  │
│ 2. ❌ Page → Repository                                                           │
│ 3. ❌ Component → Database                                                        │
│ 4. ❌ Engine 互相引用（content-engine ↔ markdown-engine 交叉 import）              │
│        → 组合权永远在 Application；Engine 各自独立、可单独测试 / 发 npm / 演进    │
└──────────────────────────────────────────────────────────────────────────────────┘
```
