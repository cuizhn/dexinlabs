

# 得心实验室课程设计规范 V1.5（Learning-driven Edition）

> **版本**：V1.5
> **定位**：Learning-first（学习优先）
> **适用范围**：所有 Lesson、Learning Unit、练习、诊断、推荐系统。
> **核心原则**：所有系统都服务于学习者，而不是服务于课程。

---

# 一、设计理念

## 1.1 我们不是在设计课程，而是在设计学习过程。

课程只是知识的载体。

真正需要设计的是：

> 学习者如何一步一步建立理解。

因此：

任何页面、任何数据库、任何系统，都必须回答一个问题：

> 它是否帮助学习者理解？

如果不能，就不应该存在。

---

## 1.2 学习遵循认知规律，而不是教材顺序。

教材顺序：

```
定义

↓

性质

↓

例题

↓

练习
```

得心实验室：

```
发现问题

↓

产生需求

↓

提出猜想

↓

建立概念

↓

形成定义

↓

理解规律

↓

迁移应用

↓

反思总结
```

学习不是接受定义。

学习是建立认知。

---

## 1.3 所有页面都根据学习状态动态变化。

页面不是固定内容。

页面应该反映学习者当前状态。

例如：

第一次学习：

```
介绍 Topic

↓

开始学习
```

学习过程中：

```
继续学习

↓

当前位置

↓

下一步建议
```

已经掌握：

```
知识网络

↓

复习建议

↓

练习巩固
```

页面属于学习者。

不是属于课程。

---

# 二、学习模型

学习模型采用以下层级。

```
Domain

↓

Topic

↓

Lesson

↓

Learning Unit

↓

Concept
```

---

## Domain

学习领域。

作用：

组织 Topic。

帮助学习者建立整体认知。

不是知识边界。

不是教材章节。

未来主要作为知识地图过滤条件存在。

---

## Topic

学习主题。

围绕一个核心问题组织。

例如：

```
函数

一次函数

二次函数

圆

概率
```

Topic 是学习入口。

也是学习状态的核心对象。

---

## Lesson

一次完整学习活动。

Lesson 不等于教材课时。

Lesson 的目标：

帮助学习者完成一次完整理解。

Lesson 可以包含一个或多个 Learning Unit。

---

## Learning Unit

一次认知推进。

Learning Unit 是教学最小单位。

不是固定长度。

不是固定数量。

每一个 Unit 都应该完成一次认知提升。

例如：

```
提出问题

↓

观察现象

↓

提出猜想

↓

验证

↓

形成理解
```

如果 Concept 较难。

可以拆分多个 Unit。

---

## Concept

Concept 是认知节点。

Concept 是理解的最小对象。

也是未来诊断系统的最小粒度。

一个 Lesson 可以涉及多个 Concept。

一个 Concept 也可能跨多个 Lesson。

Concept 之间存在依赖关系。

未来逐步形成 Concept Graph。

---

# 三、课程设计原则

每一个 Lesson 都必须回答六个问题。

---

## ① 为什么学习？

不要直接给定义。

首先建立学习需求。

学习者必须知道：

为什么需要这个知识。

---

## ② 当前真正的问题是什么？

不是老师的问题。

而是学习者的问题。

例如：

```
为什么要学习函数？

为什么需要方程？
```

课程必须围绕问题展开。

---

## ③ 真正困难在哪里？

真正困难不是知识点。

而是认知断点。

课程需要尽量找到：

为什么学习者容易不会。

---

## ④ 如何循序渐进？

不要一次讲很多。

每一次只推进一个认知。

Learning Unit 必须连续。

不能跳跃。

---

## ⑤ 如何判断真正理解？

理解不是会做题。

真正理解应该至少满足：

能够解释。

能够举例。

能够迁移。

能够解决新的问题。

---

## ⑥ 如果不会，是哪里不会？

Lesson 结束时。

必须明确：

哪些 Concept 没建立。

未来用于诊断。

---

# 四、学习过程设计

推荐采用以下学习流程。

```
为什么学习

↓

今天解决什么问题

↓

生活中的例子

↓

自己猜一猜

↓

观察规律

↓

形成概念

↓

正式定义

↓

完整例题

↓

变化例题

↓

总结规律

↓

费曼讲解

↓

少量练习
```

根据内容可调整。

但整体遵循：

从经验到抽象。

---

# 五、诊断设计原则

诊断不是考试。

诊断是帮助学习。

因此：

诊断必须：

及时。

准确。

温和。

可解释。

---

诊断重点不是：

```
不会一次函数
```

而是：

```
不会变量

不会对应关系

不会坐标

不会函数思想
```

诊断应该逐步定位真正原因。

---

推荐必须接受诊断验证。

如果推荐效果不好。

系统应自动修正推荐方向。

形成：

```
推荐

↓

学习

↓

诊断

↓

修正推荐
```

闭环。

---

# 六、学习进度设计

Progress 不只是记录完成情况。

而是记录学习过程。

未来重点关注：

当前 Topic。

当前 Lesson。

当前 Learning Unit。

学习难点。

高频错误。

停留时间。

学习记录最终用于：

帮助学习者。

帮助家长。

帮助推荐系统。

帮助诊断系统。

---

# 七、首页设计原则

首页不是课程目录。

首页是学习入口。

进入首页：

```
检查 Progress
```

没有学习记录：

```
开始学习
```

进行轻量学习了解：

选择学习阶段。

完成少量基础问题。

建立初步学习画像。

然后开始学习。

有学习记录：

```
继续学习
```

直接返回当前学习位置。

首页永远优先帮助学习。

---

# 八、知识地图设计

知识地图不是课程目录。

知识地图帮助学习者建立整体认知。

默认：

直接展示全部 Topic。

例如：

```
函数

方程

圆

三角形

概率
```

Domain 不作为主体。

仅作为过滤器。

Topic 根据学习状态动态显示：

```
已掌握

正在学习

待学习
```

知识地图帮助学习者理解：

自己在哪里。

下一步去哪里。

---

# 九、Topic 页面设计

Topic 页面根据学习状态动态变化。

待学习：

介绍 Topic。

说明学习意义。

开始探索。

学习中：

显示 Lesson。

显示当前位置。

提示继续学习。

适时推荐练习。

已掌握：

显示知识概览。

显示知识联系。

建议复习。

建议练习。

Topic 不是目录。

而是学习控制中心。

---

# 十、Lesson 页面设计

Lesson 页面强调沉浸学习。

去除干扰。

默认不显示顶部导航。

仅保留返回 Topic。

页面建议布局：

左侧：

待解决的问题。

当前目标。

右侧：

智能提示。

依据 Progress 与 Diagnosis 动态生成。

右下角：

我的理解（学习笔记）。

鼓励学习者用自己的语言表达。

未来支持关联 Topic、Concept 与相关知识，而不是传统章节导航。

---

# 十一、产品设计原则

得心实验室不是内容平台。

而是学习平台。

因此：

所有模块都必须服务学习。

```
Content

↓

Progress

↓

Diagnosis

↓

Assessment

↓

Recommendation
```

所有系统共同形成：

```
观察

↓

判断

↓

调整

↓

继续学习
```

学习不是一次完成。

而是不断修正。

不断理解。

螺旋上升。

---

# 十二、V1.5 当前阶段原则（最重要）

未来系统会包含：

* Progress Engine
* Diagnosis Engine
* Recommendation Engine
* Assessment Engine
* Concept Graph

但 **V1 阶段不追求全部实现**。

当前工作的重点只有两个：

1. 建立稳定、正确、可扩展的学习模型。
2. 编写高质量课程，在实际写作中不断验证"什么是真正的理解""什么是真正的循序渐进""什么才值得诊断"。

**架构不是最终目标。**

**课程不是最终目标。**

**真正的目标，是帮助学习者一步一步建立理解。**


我认为，**V1.5 与 V1 最大的区别只有一句话**：

> **V1 关注如何组织课程；V1.5 开始关注学习者如何真正学会。**

这也是得心实验室从"课程平台"走向"学习平台"的关键转折。
