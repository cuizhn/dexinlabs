⚠ P0 严重错误 — 导致运行时不正确行为
P0 全局 CSS typo：@media 被写成了 @markdowndia（11 处 / 6 文件）
所有响应式媒体查询的 @media 关键字被误写为 @markdowndia，这是无效的 CSS at-rule，浏览器会静默忽略整个规则块。导致全站移动端响应式布局完全失效——手机端导航菜单不切换、网格不折叠、字号不缩小。
受影响文件：
• app/pages/index.vue — L865, L896, L983（3处）
• app/components/home/FeatureGrid.vue — L775, L806, L893（3处）
• app/components/app/Header.vue — L203, L219（2处）
• app/components/app/Footer.vue — L115（1处）
• app/pages/methods.vue — L125（1处）
• app/pages/about.vue — L158（1处）
修复：全局替换 @markdowndia → @media（11 处）。这是最高优先级修复。
P0 数据字段不匹配：页面读取 description 但模型只有 summary
Chapter 和 Lesson 的数据库表和 TypeScript 模型中没有 description 字段（只有 summary），但代码中多处读取 .description。由于模型有 [key: string]: unknown 索引签名，TypeScript 不报错，但运行时值为 undefined，导致描述文字永远不显示。
受影响位置：
• app/content/services/ChapterService.ts:34 — description: l.description → 应为 l.summary
• app/pages/course/index.vue:23-24 — c.description → 应为 c.summary（章节卡片描述不显示）
• app/pages/course/[chapter]/index.vue:29-30 — lesson.description → 应为 lesson.summary（课时列表描述不显示）
修复：将上述 3 处 description 改为 summary。注意 Exercise 模型确实有 description 字段，exercise/[chapter].vue 中的用法是正确的。
● P1 功能缺陷 — 数据不显示 / 上下文断裂
P1 LessonService 预渲染了 introHtml / summaryHtml 但 Renderer 不使用
LessonService.getLessonPage() 对 intro 和 summaryText 做了 renderToHTML，生成了 introHtml 和 summaryHtml 字段返回给前端。但 Renderer.vue 只检查 bodyHtml 和 contentHtml，不认识 introHtml / summaryHtml。导致课时页面的"引言"和"总结"内容渲染了却不显示。
文件：app/content/services/LessonService.ts:42-44、app/components/content/Renderer.vue:28-34
修复方案：①在 Renderer 中增加对 introHtml/summaryHtml 的处理（用 slot 或分段渲染）；或 ②如果设计上 intro/summary 不需要单独显示，移除 LessonService 中的无效渲染以节省开销。
P1 CourseService 对每个 lesson 渲染 body/intro，但课程页不使用
CourseService.buildCoursePage() 遍历所有章节的所有课时，对每个 lesson.body 和 lesson.intro 调用 renderToHTML。但课程中心页面（course/index.vue）只用 chapter 的 title，不渲染任何 lesson 内容。这是大量无效的 Markdown 渲染（N 个课时 = N×2 次渲染），拖慢课程页加载。
文件：app/content/services/CourseService.ts:36-44
修复：移除 buildCoursePage 中对 lesson body/intro 的渲染，只保留 chapter 级别的 body 渲染（如果需要）。
P1 Repository 的 leftJoin 使用了全为 NULL 的外键
ChapterRepository.listByCourse、LessonRepository.listByChapter、ExerciseRepository.listByChapter 中使用 leftJoin(courses, eq(this.table.courseId, courses.id)) 做关联查询。但数据库中 courseId / chapterId 列全部为 NULL（数据实际通过 slug 字符串关联）。leftJoin 永远不会匹配到行，or(eq(courses.slug, ...)) 条件也永远走不到。join 是死代码，虽然功能上靠 eq(this.table.course, slug) 还能工作，但增加了无意义的 JOIN 开销和误导性。
文件：ChapterRepository.ts:34、LessonRepository.ts:33、ExerciseRepository.ts:67
修复：移除 leftJoin，直接用 where(eq(this.table.course, courseSlug)) 单条件查询。
P1 exercise 页面为获取标题调用完整 useChapterPage，触发冗余查询
exercise/[chapter].vue 调用 useChapterPage(chapterSlug) 仅为获取 chapter.title。但 useChapterPage 会请求 /api/chapter/:slug，触发完整的章节关系查询（含 lessons、exercise、siblingChapters）。练习页只需要一个标题，却拉取了整页数据。
文件：app/pages/exercise/[chapter].vue:73
修复：用轻量 useFetch('/api/chapter?course=...') 或直接从 exercise 数据中携带 chapter title，避免额外请求。
○ P2 流畅性问题 — 冗余 / 误导 / 不一致
P2 Schema 双字段冗余：外键列（全 NULL）与 slug 列并存
chapters 表同时有 courseId（外键 → courses.id）和 course（course_slug 字符串）；lessons 和 exercises 同理有 chapterId 和 chapter。relations 定义用 slug 关联，但外键约束、索引仍维护 courseId/chapterId。两套关联机制并存导致认知混乱，且 Repository 代码中 leftJoin 错误地使用了外键路径。
文件：app/database/schema.ts:45,73,101
建议：需要数据库迁移。方案一：填充外键列并改用外键关联（推荐）；方案二：移除外键列，统一用 slug。短期内确保代码一致使用 slug 路径。
P2 Schema 冗余索引：slug 上同时有 uniqueIndex 和普通 index
每个表的 slug 列上既有 uniqueIndex 又有普通 index。uniqueIndex 已自动创建索引，普通 index 完全冗余，增加写入开销。
文件：schema.ts:31-33,54-58,82-87,109-115（courses/chapters/lessons/exercises 各有）
修复：移除 idx_*_slug 普通索引，保留 idx_*_slug_unique。
P2 ExerciseRepository.listByChapter 手动列出全部字段，等同 select()
listByChapter 中用 .select({ id:..., slug:..., title:..., ... }) 手动列出了 exercises 表的所有 14 个字段，与直接 .select()（返回所有字段）完全等价。冗余代码，增加维护负担。
文件：app/content/repositories/ExerciseRepository.ts:50-65
修复：替换为 .select()，删除手动字段列表。
P2 FeatureGrid.vue 与 index.vue 大量重复 / 死 CSS
FeatureGrid.vue 包含 900+ 行 CSS，但组件模板只渲染 .landing__features 部分，.landing__hero*、.landing__course*、.landing__btn* 等大量样式是从 index.vue 复制来的死代码。同理 index.vue 中也有未使用的 .landing__features* 和 .landing__course* 样式。两个文件各自重复定义了同一批 CSS。
文件：app/components/home/FeatureGrid.vue（L65-935）、app/pages/index.vue（L162-1026）
修复：FeatureGrid 只保留 .landing__features* 相关样式；index.vue 只保留 hero/how 相关样式。重复的 .landing__btn 提取到全局 CSS。
P2 FeatureGrid.vue props 声明与本地变量同名，props 从未使用
defineProps({ features }) 声明了 features prop，但同时 const features = [...] 定义了同名本地数组。模板中 v-for="feat in features" 解析到本地变量，props 从未被父组件传入也未被使用。
文件：app/components/home/FeatureGrid.vue:23-25
修复：移除 defineProps 声明，或改为从父组件传入数据。
P2 study.vue 内容与技术栈不符
study.vue 硬编码的 Markdown 测试文本提到 "Morph-MD 组件库"、"marked.lexer + MDAST"、"Prism.js 代码高亮"、"Mermaid 图表" 等，但项目实际使用的是 unified + remark + rehype-katex，没有 marked/Prism/Mermaid。内容是旧技术栈的残留。
文件：app/pages/study.vue:21-56
修复：更新测试内容使其与实际技术栈一致，或移除此测试页面（Header 导航中已无此链接）。
P2 exercise/index.get.ts 有 DATABASE_URL 检查但其他 API 没有
只有 exercise/index.get.ts 在入口处检查了 process.env.DATABASE_URL，其他 API（course/chapter/lesson）都没有。行为不一致，且实际数据库连接错误会在 getDb() 中抛出，这个检查是冗余的。
文件：server/api/exercise/index.get.ts:5-10
修复：移除该检查（由 getDb() 统一处理），或所有 API 统一添加。
